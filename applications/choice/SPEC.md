# M1 Choice Model Specification

## Overview

**M1 Choice Model** is a two-tower neural architecture that predicts consumer choices from their Consumer Digital Twin (CDT) embedding and product features. It transforms the CDT from "who is this person?" to "what will they choose?" by learning to predict purchase decisions.

**Architecture:** CDT[128] + Product[D] → Consumer Tower + Product Tower → Joint Layer → P(choose_this_product)

**Success Criteria:**
1. **CDT lift ≥ 0.05 AUC** over no-CDT baselines (product-features-only, persona-id one-hot)
2. **AUC ≥ 0.65** on held-out participants (floor, subordinate to lift)
3. **Brier ≤ 0.25** and **calibration slope ∈ [0.8, 1.2]**

## Purpose

The M1 Choice Model serves as the foundational capability for:
- **L1 (Choice Prediction):** Predict which products a consumer will choose
- **L2 (Ranking):** Rank products by predicted choice probability  
- **M2 (Market Simulation):** Simulate market outcomes under different conditions

**Critical innovation:** Uses trace encoder embeddings (decision process) to predict choices, enabling CDT lift over product features alone.

## Architecture

### Two-Tower Design

```python
Consumer Tower:  CDT[128] → Linear(128, 64) → ReLU → Dropout(0.1) → [64]
Product Tower:   Product[D] → Linear(D, 64) → ReLU → Dropout(0.1) → [64]
Joint Layer:      Concat[128] → Linear(128, 1) → Sigmoid → P(choose)
```

**Parameters (default):**
- `cdt_dim`: 128 (frozen fusion embedding dimension)
- `product_dim`: 8 (the full §0.1 board: price, brand, quality, warranty, rating, features, availability, design — bead 6ca). A trial's inspected `displayed_attributes` is a prefix of this vector, so the product tower sees every attribute that can drive a choice, not just price + quality.
- `hidden_dim`: 64 (tower output dimension)
- `dropout`: 0.1 (regularization)

### Why Two-Tower?

1. **Modularity:** Consumer and product encoding are learned separately
2. **Interpretability:** Ablation studies can isolate CDT contribution
3. **Flexibility:** Product tower can be extended with embeddings (brand, category)
4. **Lift gate:** Direct comparison between CDT vs. no-CDT models

## Data Pipeline

### Input Sources

1. **ChoiceSets:** `data/synthetic/choice_sets.jsonl`
   - Trial-level choice records with alternatives and displayed attributes
   - Generated by trace simulator with preference-driven choice logic

2. **CDT Cache:** `applications/_cache/cdt_embeddings.parquet`
   - Pre-computed 128-dim CDT embeddings from frozen fusion encoder
   - Indexed by participant_id

3. **Product Catalog:** `data/synthetic/products.jsonl`
   - Master product catalog with category, brand, attributes
   - Referenced by ChoiceSet alternative_products

### Training Table Construction

```python
# For each ChoiceSet:
# 1. Load CDT embedding for participant_id
# 2. For each alternative (N alternatives per trial):
#    - Extract product features from displayed_attributes
#    - Create row: (participant_id, cdt_embedding, product_features, chosen)
#    - Binary label: chosen = (slot == chosen_alternative)

Result: N rows per trial (1 chosen + N-1 rejected)
```

**Output:** `applications/choice/choice_training.parquet`

**Columns:**
- `participant_id`: str
- `cdt_embedding`: list[float] (128-dim)
- `product_features`: list[float] (8-dim §0.1 board encoding of the product)
- `chosen`: bool (1 = chosen, 0 = rejected)

## Training Procedure

### Participant-Level Split

**Critical:** Use participant-level 70/30 split, NOT trial-level split.

**Why:** Trial-level splits leak participant identity (same consumer appears in both train and val). Participant-level splits test generalization to new consumers.

```python
participants = df["participant_id"].unique()
train_pids, val_pids = train_test_split(participants, test_size=0.3, random_state=42)

train_df = df[df["participant_id"].isin(train_pids)]
val_df = df[df["participant_id"].isin(val_pids)]
```

### Loss Function

**Binary Cross-Entropy (BCE):**
```python
criterion = nn.BCELoss()
loss = criterion(probs, labels)  # probs: P(choose), labels: {0, 1}
```

**Why BCE:** Choice prediction is binary classification (chosen vs. rejected). BCE directly optimizes choice probability calibration.

### Optimization

**Optimizer:** AdamW with weight decay
```python
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=1e-4)
```

**Hyperparameters:**
- `lr`: 1e-3 (learning rate)
- `weight_decay`: 1e-4 (L2 regularization)
- `batch_size`: 256
- `epochs`: 50 with early stopping

### Early Stopping

**Patience:** 10 epochs on validation AUC

```python
if val_auc > best_val_auc:
    best_val_auc = val_auc
    patience_counter = 0
    torch.save(model.state_dict(), "choice_head.pt")
else:
    patience_counter += 1
    if patience_counter >= max_patience:
        break  # Early stopping
```

## Evaluation

### Metrics

1. **AUC (Area Under ROC Curve):** Choice prediction accuracy
2. **Brier Score:** Probability calibration (lower = better calibrated)
3. **Calibration Slope:** Reliability of probability estimates

### CDT Lift Gate

**Critical requirement:** M1 must beat no-CDT baselines by ≥ 0.05 AUC

**Baseline 1: Product-features-only**
- Logistic regression on product features only (no CDT)
- Represents "demographics + product attributes" baseline

**Baseline 2: Persona-id one-hot**  
- Logistic regression on persona ID only (no CDT)
- Represents "archetype membership" baseline

**Lift calculation:**
```python
lift_over_product = m1_auc - product_only_auc
lift_over_persona = m1_auc - persona_auc

# Gate check
if lift_over_product < 0.05:
    print("❌ M1 FAILED: CDT lift < 0.05")
```

**Why lift gate matters:** High AUC can be achieved by persona leakage. Lift over product features proves *decision process* (trace) predicts choice — the core innovation.

### Calibration Check

**Brier score:** Mean squared error of predicted probabilities
```python
brier = brier_score_loss(true_labels, pred_probs)
# Threshold: ≤ 0.25
```

**Calibration slope:** Regression slope of calibration curve
```python
fraction_of_positives, mean_predicted_value = calibration_curve(true_labels, pred_probs)
slope, _ = np.polyfit(mean_predicted_value, fraction_of_positives, 1)
# Threshold: [0.8, 1.2]
```

**Why calibration:** Well-calibrated probabilities are critical for downstream applications (ranking, market simulation). Overconfident or underconfident predictions mislead decision-makers.

## Success Criteria

> Reframed in bead 16r. The binding criterion is **lift over a product-only
> baseline plus oracle-attainability**, not an absolute AUC floor. The absolute
> 0.65 floor was unreachable by construction when the generator's choice labels
> were near-uniform (oracle ceiling 0.648); judging M1 against it was the M1
> post-mortem's core misdiagnosis. The oracle (the choice set's own generative
> probabilities) is the Bayes-optimal predictor and bounds what any model can
> achieve — M1 must clear 85% of it AND beat product-only by ≥ 0.05.

### Tier 1: Lift Gate (Critical) — PASS requires both

- ✅ **`lift_over_product ≥ 0.05`** — `M1_AUC − product_only_AUC`, where
  `product_only` is a LogisticRegression on `product_features` alone. Proves the
  decision process (trace → CDT) predicts choice better than product attributes.
- ✅ **`M1_AUC ≥ 0.85 · oracle_AUC`** — M1 reaches 85% of the Bayes-optimal
  ceiling (the choice set's own `choice_probabilities`). Rules out "good AUC that
  is still far below what the labels allow".

All predictors are scored on the SAME participant-level val split used to train
M1 (`train_test_split(participants, test_size=0.3, random_state=42)`). The oracle
is a lookup, not fit, so there is no leakage.

### Tier 2: Calibration (Informational)

- ℹ️ **Brier ≤ 0.25** (probability calibration)
- ℹ️ **Calibration slope ∈ [0.8, 1.2]** (reliability)
- ℹ️ Absolute `AUC ≥ 0.65` is printed for context but is **not** part of the gate.

### Tier 3: Documentation

- ✅ **SPEC.md** written (this document)
- ✅ Module boundary invariant respected (no training imports, reads from cache)

## Module Boundaries

### Invariant

**Applications must NOT import training modules.** M1 reads from pre-computed CDT cache, not from encoders or generator.

**Why:**
1. **Separation of concerns:** Applications use CDT, don't create it
2. **Reproducibility:** Cache ensures consistent embeddings
3. **Performance:** Avoids recomputing expensive embeddings
4. **Safety:** Prevents encoder drift during application development

### Cache Protocol

**Location:** `applications/_cache/cdt_embeddings.parquet`

**Format:** Parquet table with columns:
- `participant_id`: str
- `cdt`: list[float] (128-dim embedding)
- `persona`: str (for ablation studies)

**Generation:** Run fusion encoder on synthetic dataset
```bash
uv run python -m fusion.predict --output applications/_cache/cdt_embeddings.parquet
```

## Failure Modes

### Lift Gate Failure

**Symptom:** CDT lift < 0.05 despite good AUC

**Root causes:**
1. **Weak trace-choice coupling:** Trace encoder doesn't capture decision process
2. **Product features too predictive:** Price/quality dominate choice variance
3. **Insufficient CDT signal:** Fusion encoder didn't learn trace representation

**Mitigation:**
- Verify trace-choice coupling with `evaluation/verify_trace_choice_coupling.py`
- Tune `inspection_bonus` coefficient in choice utility calculation
- Check fusion encoder probes (strategy recovery ≥ 95%)

### Calibration Failure

**Symptom:** Brier > 0.25 or slope outside [0.8, 1.2]

**Root causes:**
1. **Overfitting:** Model memorized training set
2. **Underfitting:** Model too simple to capture patterns
3. **Data imbalance:** Chosen/rejected ratio skewed

**Mitigation:**
- Adjust dropout (increase if overfitting, decrease if underfitting)
- Verify participant-level split (no data leakage)
- Check choice set balance (should be ~1:N chosen-to-rejected ratio)

## Future Work

### L2 (Ranking)

**Extension:** Rank products by predicted choice probability

**Implementation:**
```python
def rank_products(cdt_embedding, product_alternatives):
    probs = []
    for product in product_alternatives:
        prob = model(cdt_embedding, product.features)
        probs.append(prob)
    
    ranking = sorted(zip(product_alternatives, probs), key=lambda x: -x[1])
    return ranking
```
